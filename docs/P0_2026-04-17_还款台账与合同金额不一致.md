## 3. 根因分析

### 3.1 直接原因

`repayment_plan_item.amount`（台账）与 `contract.meta.rentPerPeriod`（合同冻结值）之间**没有一致性约束**：

- 合同签署时通过 `AdminContractController.buildOrderMeta()`
  将 `max(amount)` 取自 plan 写入 `meta.rentPerPeriod`，固化为**法律层面的每期金额**
- 但 `repayment_plan_item.amount` 会被多处入口**按最新 `product.rent_*` 重新计算**：
  - `H5OrderController.create` / `update`
  - `AdminOrderController.adjustPrice`
  - `AdminRepaymentController.generatePlan?force=true`
- 其中只有 `adjustPrice` 会连带作废合同；其余路径**不做合同状态检查**即可覆写 plan

### 3.2 触发条件

- 后台（`AdminProductController.upsert`）曾将若干车型的 `rent_per_cycle` / `rent_with_battery` / `rent_without_battery` 调整过价格
- 随后有**脚本 / 数据迁移 / 管理员手动点击"重新生成还款计划"**等操作，按当时的产品价格重建了 plan，覆盖了合同签约时的金额

### 3.3 为何长期没被发现

- 台账与合同是两张相互独立的"真相源"，前端没有做交叉校验
- 客户端看到的台账金额异常，通常要人工对账才会发现
- `order_log` 没有捕捉到覆写 plan 的 `PLAN_GENERATED`，说明覆写大概率来自绕过接口的脚本或直接 SQL，而非正规后台接口

## 4. 修复方案（已全部上线）

### 4.1 代码：保护层（拒绝再改）

| 文件 | 修改 |
|------|------|
| `services/api/.../H5OrderController.java` | 新增 `assertNoSignedContract(orderId)`，在 `update` 前调用。如订单有 `SIGNED` 合同则抛 400 |
| `services/api/.../AdminRepaymentController.java` | `generatePlan`（含 `force=true`）新增合同 SIGNED 检查，拒绝覆盖 |

> `AdminOrderController.adjustPrice` 原本只允许 `PENDING_REVIEW`，本身就安全，未改。

### 4.2 代码：快照层（显示以合同为准）

| 文件 | 修改 |
|------|------|
| `services/api/.../service/OrderEnricher.java` | 新增 `extractContractRentPerPeriod(Contract)`；`enrich()` 读取 plan 时若合同已 SIGNED，自动用 `meta.rentPerPeriod` 覆盖每期 amount 输出给前端。这样即使 DB 里 plan 被再次污染，前端展示的台账、剩余应还都与合同一致 |

### 4.3 数据：一次性修正 + 备份 + 回滚

- 执行 SQL：
  ```sql
  UPDATE repayment_plan_item rpi
  JOIN contract c ON c.order_id = rpi.order_id
  SET rpi.amount = CAST(JSON_UNQUOTE(JSON_EXTRACT(c.meta,'$.rentPerPeriod')) AS UNSIGNED)
  WHERE c.contract_type='ORDER' AND c.status='SIGNED'
    AND JSON_EXTRACT(c.meta,'$.rentPerPeriod') IS NOT NULL
    AND rpi.amount <> CAST(JSON_UNQUOTE(JSON_EXTRACT(c.meta,'$.rentPerPeriod')) AS UNSIGNED)
    AND rpi.order_id IN (plan 均匀的订单子集);
  ```
- 共刷新 **186 行**（杨建单独 6 行 + 24 单批量 180 行）
- **备份表**：`repayment_plan_item_backup_20260417`（全部被改行的原值已存）
- **回滚 SQL**：
  ```sql
  UPDATE repayment_plan_item rpi
  JOIN repayment_plan_item_backup_20260417 b ON b.plan_id = rpi.id
  SET rpi.amount = b.amount;
  ```

## 10. 经验教训

1. **跨表的"法律真相源"必须做强一致性校验**。合同是法律文件，一旦签署，与之绑定的金融字段必须有跨系统的约束或至少每日对账
2. **凡是写操作的入口必须检查业务状态**。合同 SIGNED、已发货、已入账等都应是修改拦截点
3. **产品价格调整是高风险操作**。改价前应检查有多少已签合同依赖当前价格，并至少产出影响评估报告
4. **"修复脚本"和"直连 SQL"必须留痕**。本次根因之一是绕开接口直接改数据，没有任何审计，事后无法确定是哪个脚本触发
